# 2026-06-09: 바이트부터 TCP 메시지 경계까지

> 목표: `00 00 00 2A` 같은 값을 보고 겁먹지 않고, TCP에서 왜 메시지 경계 약속이 필요한지 이해한다.

## 오늘의 결론

```text
네트워크로 오가는 데이터는 결국 바이트다.
문자열도 UTF-8 같은 인코딩을 거쳐 바이트 배열이 된다.
TCP는 메시지 하나의 끝을 알려주지 않는다.
그래서 클라이언트와 서버가 메시지 경계 규칙을 프로토콜로 약속해야 한다.
```

## 한눈에 보기

| 개념 | 오늘 이해한 내용 |
| --- | --- |
| `00 00 00 2A` | 숫자 42를 4바이트로 표현한 값 |
| UTF-8 | 문자열을 바이트로 바꾸는 인코딩 방식 |
| TCP stream | 데이터가 메시지 단위가 아니라 바이트 흐름으로 전달됨 |
| protocol | 서버와 클라이언트가 데이터를 해석하기 위해 정한 약속 |
| length header | body의 바이트 길이를 헤더에 써서 메시지 경계를 알리는 방식 |

## 1. 오늘 이걸 공부한 이유

백엔드 개발을 하다 보면 프레임워크 코드는 어느 정도 따라 만들 수 있다. 하지만 네트워크나 파일, 메모리처럼 아래 계층의 개념이 부족하면 다음 질문에서 막히기 쉽다.

```text
TCP에서 메시지 하나가 어디서 끝나는지 어떻게 알까?
헤더에 길이를 넣는다는 건 정확히 무슨 뜻일까?
문자열은 네트워크로 갈 때 어떤 값으로 변할까?
```

이 질문에 답하려면 먼저 컴퓨터가 데이터를 바이트로 다룬다는 사실을 알아야 한다.

## 2. bit와 byte

`bit`는 0 또는 1 하나를 의미한다.

```text
0
1
```

`byte`는 8개의 bit를 묶은 단위다.

```text
00000000
11111111
```

8bit로 표현할 수 있는 값은 0부터 255까지다.

```text
00000000 = 0
00000001 = 1
00000010 = 2
11111111 = 255
```

네트워크에서 데이터를 보낸다는 말은 결국 바이트 배열을 보낸다는 뜻이다.

## 3. 16진수를 쓰는 이유

2진수는 컴퓨터 입장에서는 자연스럽지만 사람이 읽기에는 길다.

```text
00101010
```

이 값을 16진수로 쓰면 짧아진다.

```text
2A
```

16진수는 한 자리로 0부터 15까지 표현한다.

| 10진수 | 16진수 |
| --- | --- |
| 10 | A |
| 11 | B |
| 12 | C |
| 13 | D |
| 14 | E |
| 15 | F |

`2A`를 10진수로 바꾸면 다음과 같다.

```text
2A(hex)
= 2 * 16 + 10
= 42
```

그래서 `0x2A`는 10진수 `42`다.

## 4. `00 00 00 2A`는 무엇인가

`00 00 00 2A`는 4바이트 정수로 숫자 42를 표현한 예시다.

```text
00 00 00 2A
```

앞의 0들은 자리수를 맞추기 위한 값이고, 마지막 `2A`가 실제로 42를 의미한다.

```text
0x0000002A = 42
```

네트워크 프로토콜에서 앞 4바이트를 body 길이로 쓰기로 약속했다면, 이 값은 이렇게 해석된다.

```text
뒤에 42바이트의 body가 온다.
```

중요한 점은 서버가 body를 보고 길이를 추측하는 것이 아니라는 점이다. 클라이언트가 body 길이를 먼저 계산해서 헤더에 써서 보낸다.

## 5. 문자열은 네트워크에서 바이트가 된다

문자열도 네트워크로 보낼 때는 바이트로 변환된다.

Java에서는 보통 UTF-8 인코딩을 사용한다.

```java
byte[] bytes = text.getBytes(StandardCharsets.UTF_8);
```

영어 숫자와 기호는 보통 한 글자가 1바이트다.

```text
A -> 41
B -> 42
```

한글은 UTF-8에서 보통 한 글자가 3바이트다.

```text
가 -> EA B0 80
```

그래서 문자열 길이와 바이트 길이는 항상 같지 않다.

```text
"ABC".length() = 3
"ABC".getBytes(UTF-8).length = 3

"가".length() = 1
"가".getBytes(UTF-8).length = 3
```

길이 헤더를 만들 때는 문자 개수가 아니라 실제 body의 바이트 길이를 넣어야 한다.

## 6. TCP는 메시지가 아니라 stream이다

TCP에서 가장 중요한 문장은 이것이다.

```text
TCP는 메시지 단위를 보장하지 않고, 바이트 stream만 보장한다.
```

클라이언트가 메시지를 두 번 보냈다고 해서 서버가 두 번으로 읽는다는 보장은 없다.

```text
client write 1: message-1
client write 2: message-2
```

서버에서는 이렇게 읽힐 수 있다.

```text
read 1: message-1message-2
```

또는 이렇게 쪼개질 수도 있다.

```text
read 1: mess
read 2: age-1message
read 3: -2
```

그래서 TCP 서버에서는 메시지 경계를 알기 위한 규칙이 필요하다.

## 7. header/body와 protocol

프로토콜은 서버와 클라이언트가 데이터를 해석하기 위해 정한 약속이다.

길이 기반 프로토콜은 보통 이렇게 생겼다.

```text
[Header][Body]
```

가장 단순한 형태는 다음과 같다.

```text
[4 bytes body length][body bytes]
```

예를 들어 body가 42바이트라면 클라이언트는 앞에 4바이트 길이 헤더를 붙인다.

```text
[00 00 00 2A][42 bytes body]
```

서버는 앞 4바이트를 읽고 `42`로 해석한 뒤, 뒤에서 42바이트가 다 모일 때까지 기다린다. 다 모이면 body 하나를 완성된 메시지로 처리한다.

이 방식은 body 안에 줄바꿈이 있든, XML 태그가 길든, JSON 필드가 많든 상관없이 메시지를 구분할 수 있다.

## 8. 면접에서 설명할 수 있는 문장

TCP 메시지 구분에 대해서는 이렇게 설명할 수 있다.

> TCP는 stream 기반이라 애플리케이션 메시지의 경계를 보장하지 않습니다. 그래서 클라이언트와 서버가 메시지 경계 규칙을 프로토콜로 약속해야 합니다. 예를 들어 줄바꿈을 끝으로 정할 수도 있고, 더 안정적으로는 헤더에 body length를 넣어서 서버가 그 길이만큼 body를 모은 뒤 메시지 하나로 처리하게 할 수 있습니다.

길이 헤더에 대해서는 이렇게 설명할 수 있다.

> 길이 헤더는 TCP가 자동으로 붙여주는 값이 아니라 클라이언트가 body의 바이트 길이를 계산해서 앞에 붙여 보내는 값입니다. 서버는 그 헤더를 읽고 body가 몇 바이트인지 알 수 있습니다.

UTF-8에 대해서는 이렇게 설명할 수 있다.

> 네트워크로 문자열을 보낼 때는 문자열이 인코딩을 거쳐 바이트 배열로 변환됩니다. 그래서 메시지 길이를 계산할 때는 문자 수가 아니라 인코딩된 바이트 수를 기준으로 해야 합니다.
